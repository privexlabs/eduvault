# Soroban Contract Architecture Note

Status: accepted for the first Stellar-native implementation pass.

This note defines the contract boundary, storage ownership model, event contract, and trust assumptions for EduVault's Soroban milestone. It is the canonical reference for issue #30 and should be treated as the source of truth for later contract, backend, and frontend work.

## Goals

- register educational materials with immutable creator and rights anchors
- accept payment in approved Stellar assets without price oracles
- mint a durable purchase entitlement that the app can verify
- distribute creator and platform payouts atomically and audibly
- give the backend indexer and frontend wallet flow a stable interface

## Non-goals for v1

- on-chain file storage
- secondary resale or entitlement transfer
- on-chain refunds, revocation, or expiration
- dynamic FX pricing or oracle-based conversions
- arbitrary admin upgrades to live production contracts

## Decision Summary

- Use two contracts: `MaterialRegistry` and `PurchaseManager`.
- `MaterialRegistry` is the source of truth for creator-owned listing state.
- `PurchaseManager` is the source of truth for payment settlement, platform config, and buyer entitlements.
- Assets are represented only by Stellar Asset Contract addresses. Human-readable asset code and issuer stay off-chain.
- Each material stores explicit per-asset price quotes. EduVault does not use exchange-rate oracles.
- Entitlements are perpetual, non-transferable, and unique per `(material_id, buyer)` pair.
- Production deployments are migration-driven, not upgradeable in place.

## Contract Boundary

### `MaterialRegistry`

Responsibilities:

- register a new material under an immutable creator address
- anchor `metadata_hash` and `rights_hash`
- store material status, accepted-asset quotes, and creator-defined payout splits
- expose read methods for frontend and backend consumers
- emit material lifecycle events

Must not:

- receive buyer payments
- distribute funds
- write entitlement state

### `PurchaseManager`

Responsibilities:

- hold global platform config such as treasury recipient, pause flag, and allowed assets
- validate that a material is purchasable
- collect payment from the buyer and route payouts atomically
- write entitlement state
- emit purchase and payout events

Must not:

- mutate creator identity or rights hashes
- rewrite an existing entitlement except through a future explicit migration

### Boundary rule

`MaterialRegistry` owns seller-authored listing data. `PurchaseManager` owns platform-authored configuration and buyer entitlement data. Backend services may cache both, but neither cache is authoritative.

## Storage Model

### Identifier conventions

- `material_id`: `BytesN<32>`, generated by `MaterialRegistry` as `sha256(creator || creator_nonce)`.
- `asset`: Soroban contract address for the Stellar Asset Contract.
- `rights_hash`: `BytesN<32>` hash of the canonical off-chain license or rights payload.
- `metadata_hash`: `BytesN<32>` hash of the canonical metadata JSON stored off-chain.

### `MaterialRegistry` state

`MaterialRecord`

- `material_id`
- `creator: Address`
- `metadata_uri: String`
- `metadata_hash: BytesN<32>`
- `rights_hash: BytesN<32>`
- `status: MaterialStatus`
- `quotes: Vec<AssetQuote>`
- `payout_shares: Vec<PayoutShare>`
- `created_ledger: u32`
- `updated_ledger: u32`

`AssetQuote`

- `asset: Address`
- `amount: i128`

`PayoutShare`

- `recipient: Address`
- `share_bps: u32`

`MaterialStatus`

- `Active`
- `Paused`
- `Archived`

Additional keys:

- `creator_nonce(Address) -> u64`
- `material(BytesN<32>) -> MaterialRecord`

There is intentionally no on-chain pagination index for marketplace browsing. Material lists, creator dashboards, and search remain off-chain indexer concerns.

### `PurchaseManager` state

`PlatformConfig`

- `registry: Address`
- `treasury: Address`
- `platform_fee_bps: u32`
- `paused: bool`

`EntitlementRecord`

- `material_id: BytesN<32>`
- `buyer: Address`
- `active: bool`
- `purchase_id: u64`
- `asset: Address`
- `amount: i128`
- `granted_ledger: u32`

Additional keys:

- `platform_config -> PlatformConfig`
- `allowed_asset(Address) -> bool`
- `purchase_nonce -> u64`
- `entitlement((BytesN<32>, Address)) -> EntitlementRecord`

Purchase history is event-derived in v1. The chain stores the current entitlement record, while the indexer builds the historical purchases table and payout audit trail off emitted events.

## Invariants

- `material_id` is unique and never changes.
- `creator`, `metadata_hash`, and `rights_hash` are immutable after registration.
- `quotes` must be non-empty for an `Active` material.
- Every quote amount must be positive.
- An asset may appear only once inside a material's `quotes`.
- `payout_shares` must be non-empty, use unique recipients, and sum to exactly `10_000` basis points.
- `platform_fee_bps` must be between `0` and `1_000` inclusive. A higher cap requires a new architecture decision.
- A purchase can succeed only when the material is `Active`, the selected asset exists in the material quotes, the asset is currently allowed globally, and the buyer does not already hold an active entitlement.
- Payment routing and entitlement writes are atomic. If any transfer or validation fails, the whole transaction reverts and no entitlement is granted.
- Pausing or archiving a material blocks new purchases but does not revoke existing entitlements.
- Creator ownership is not transferable in v1.

Recommended implementation limits:

- `MAX_QUOTES_PER_MATERIAL = 4`
- `MAX_PAYOUT_RECIPIENTS = 5`

These caps keep storage bounded and event payloads indexable.

## Accepted-Asset Model

EduVault uses a two-layer asset rule:

1. `PurchaseManager` maintains a platform allowlist of supported Stellar Asset Contract addresses.
2. Each `MaterialRecord` stores one or more explicit `AssetQuote` entries.

The effective accepted-asset set for a purchase is the intersection of those two layers.

Why this model:

- creators can choose whether to accept XLM, USDC, or another approved asset
- the platform can disable a risky or unsupported asset without rewriting every material
- explicit prices per asset remove the need for oracles and protect the purchase path from stale exchange assumptions

## Payout Rules

Payout calculation for a successful purchase is:

1. `gross = selected_quote.amount`
2. `platform_fee = gross * platform_fee_bps / 10_000`
3. `seller_net = gross - platform_fee`
4. split `seller_net` across `payout_shares`

Rules:

- the treasury receives the platform fee
- `payout_shares` split only the seller net, not the gross amount
- payout recipients are not required to equal the immutable creator address
- all payout transfers happen in the same transaction that grants entitlement
- the contract should not retain user funds after a successful purchase

This keeps creator attribution separate from payout routing, which matters for publisher accounts, collaborators, or institution-managed disbursement.

## Minimum Contract Interface

Frontend and backend code can depend on the following minimum read and write surface.

### `MaterialRegistry`

```text
register_material(
  creator,
  metadata_uri,
  metadata_hash,
  rights_hash,
  quotes,
  payout_shares
) -> Result<material_id, RegistryError>

update_sale_terms(material_id, quotes, payout_shares) -> Result<(), RegistryError>

set_material_status(material_id, status) -> Result<(), RegistryError>

get_material(material_id) -> Result<MaterialRecord, RegistryError>

get_quote(material_id, asset) -> Result<Option<AssetQuote>, RegistryError>
```

Auth expectations:

- `register_material`, `update_sale_terms`, and `set_material_status` require creator auth
- only the original creator may mutate sale terms or status

### `PurchaseManager`

```text
purchase(material_id, asset, expected_amount) -> purchase_id

has_entitlement(material_id, buyer) -> bool

get_entitlement(material_id, buyer) -> Option<EntitlementRecord>

get_platform_config() -> PlatformConfig

is_asset_allowed(asset) -> bool

set_asset_allowed(asset, enabled)

set_platform_config(treasury, platform_fee_bps, paused)
```

Auth expectations:

- `purchase` requires buyer auth and must verify `expected_amount` against the current quote to prevent stale-UI purchases
- `set_asset_allowed` and `set_platform_config` require platform admin auth

## Entitlement Query Model

The authoritative post-purchase check is `has_entitlement(material_id, buyer)` or `get_entitlement(material_id, buyer)`.

Operationally:

- frontend may optimistically show success after a confirmed purchase transaction, then confirm by reading the contract
- backend download gating should use `entitlement_cache` for fast reads but must fall back to a direct contract query when cache state is missing or stale
- indexer-derived `purchases` and `entitlement_cache` records remain derived data and must be recoverable from chain events plus contract reads

In other words, entitlement is stored on-chain and cached off-chain. It is not purely event-derived.

## Event Contract

Soroban events should be normalized by the indexer into the following stable event types.

### `material.registered`

Emitter: `MaterialRegistry`

Required fields:

- `materialId`
- `creator`
- `metadataUri`
- `metadataHash`
- `rightsHash`
- `status`
- `quotes`
- `payoutShares`
- `ledger`

Use:

- seed the canonical chain linkage for a material in MongoDB
- drive creator-facing transaction success UI

### `material.sale_terms_updated`

Emitter: `MaterialRegistry`

Required fields:

- `materialId`
- `creator`
- `quotes`
- `payoutShares`
- `status`
- `ledger`

Use:

- refresh marketplace price and asset caches
- keep frontend quotes honest without reading every contract on every page load

### `material.status_updated`

Emitter: `MaterialRegistry`

Required fields:

- `materialId`
- `status`
- `ledger`

Use:

- hide archived or paused materials from purchase flows while preserving entitlement reads

### `purchase.completed`

Emitter: `PurchaseManager`

Required fields:

- `purchaseId`
- `materialId`
- `buyerAddress`
- `sellerAddress`
- `asset`
- `amount`
- `platformFee`
- `sellerNetAmount`
- `entitlementActive`
- `ledger`

Use:

- create the backend `purchases` record
- upsert `entitlement_cache`
- power the frontend purchase success state

### `payout.distributed`

Emitter: `PurchaseManager`

Required fields:

- `purchaseId`
- `materialId`
- `recipient`
- `role`
- `asset`
- `amount`
- `ledger`

Use:

- payout audit trail
- creator earnings history
- treasury reconciliation

### `admin.asset_policy_updated`

Emitter: `PurchaseManager`

Required fields:

- `asset`
- `enabled`
- `ledger`

### `admin.platform_config_updated`

Emitter: `PurchaseManager`

Required fields:

- `treasury`
- `platformFeeBps`
- `paused`
- `ledger`

Notes:

- the normalized event names above are the backend contract. Raw Soroban topics can vary, but the indexer must map them into these stable shapes
- `purchase.completed` and `payout.distributed` must be idempotent under repeated indexing
- backend code should continue using stable event ids derived from ledger, tx hash, topic, and index

## Admin Assumptions

Platform admin powers in v1:

- set the allowed-asset list
- set treasury recipient
- set platform fee within the documented cap
- pause or unpause new purchases globally

Platform admin may not:

- modify creator ownership
- change `rights_hash` or `metadata_hash`
- revoke or rewrite active entitlements
- confiscate in-flight purchase value outside the documented fee path

Creator powers in v1:

- register materials
- update sale terms
- pause or archive a material

Creator may not:

- transfer ownership of a material to another account
- backdate entitlements
- purchase their way around asset or status validation

## Upgradeability and Migration Stance

EduVault should be migration-driven.

That means:

- deployed production contracts are treated as immutable for their version
- breaking changes ship as a new contract pair, for example `MaterialRegistryV2` and `PurchaseManagerV2`
- the web app and backend may temporarily read multiple contract versions during migration
- existing entitlements on older versions remain valid and queryable until an explicit migration plan says otherwise

This choice keeps the trust model simple. Users do not need to trust an admin key that can replace contract logic after purchase.

## Security and Failure Assumptions

- file bytes and private metadata stay off-chain; only hashes and references are anchored on-chain
- every state-changing path must use Soroban auth checks for the relevant actor
- asset transfer failures revert the entire purchase
- there is no silent fallback to an alternate asset or price
- indexer lag is expected; indexer inconsistency must not block direct entitlement verification
- Stellar finality is treated as strong enough for entitlement indexing, but off-chain processing must remain idempotent
- v1 does not support refunds or entitlement revocation; any future addition changes the trust model and needs a follow-up ADR

## Backend Alignment

This note aligns with `docs/backend-contracts.md` as follows:

- `materials.materialId` stores the hex-encoded `material_id`
- `materials.chainContractId` identifies the registry or purchase contract that emitted the canonical event
- `purchases` is a derived cache built from `purchase.completed`
- `entitlement_cache` is a derived cache built from `purchase.completed` plus direct contract verification when needed
- `sync_events` remains the idempotency log for normalized Soroban events

Issue #22 should treat the purchase call and post-purchase entitlement reads in this note as the frontend wallet contract. Issue #26 should treat the normalized event types in this note as the backend indexing contract.
